# Besu private network

This folder contains the reproducible configuration for the demo Hyperledger Besu network.

## Topology

| Node | Role | RPC exposed to host | Metrics |
|---|---|---|---|
| `validator1` | QBFT validator and bootnode | No | Internal Docker network |
| `validator2` | QBFT validator | No | Internal Docker network |
| `validator3` | QBFT validator | No | Internal Docker network |
| `validator4` | QBFT validator | No | Internal Docker network |
| `rpcnode` | Non-validator RPC node | `localhost:8545` | Internal Docker network |

## Generated material

`make besu-generate` creates local development node keys, `genesis.json`, bootnode data and local permissioning files under `blockchain/besu/generated/`.

Generated key material is development-only and is intentionally ignored by Git. It must not be reused for citizens, officers, issuers or production-like environments.

## Security posture

- QBFT validators use node keys generated by Besu.
- Functional application accounts are separate and will be created in the contracts/deployment phase.
- JSON-RPC is exposed only by `rpcnode`.
- RPC namespaces are limited to `ETH`, `NET`, `WEB3` and `QBFT`.
- `ADMIN`, `DEBUG`, `TRACE` and `PERM` are not exposed on the host-facing RPC endpoint.
- Gas is free for the private network: `zeroBaseFee=true` in genesis and `min-gas-price=0` on every node.

## Zero-gas-price transactions and the transaction pool

Besu 26.6.1's default **layered** transaction pool was observed to accept zero-gas-price
transactions (`gasPrice=0`) into the pool — visible as `1 pending` in validator logs — but
never select them for a produced block, indefinitely: `Produced empty block #N / 0 tx / 1
pending`. This reproduced both for zero-balance and funded sender accounts, so it is a
layered-pool promotion/selection issue (Besu enabled a "balance check by default" in the
layered pool since 26.4.0, and the community has open reports of transactions permanently
stuck in the layered pool's sparse tier, e.g. hyperledger/besu#9598).

Mitigation applied here: every node config generated by
`scripts/besu-generate-network.sh` sets `tx-pool="legacy"` (the simpler, pre-layered
sequenced pool), which mined zero-gas-price test transactions reliably and repeatably in
manual testing, both from a funded account and from a brand-new zero-balance account.
Revisit this pin whenever `BESU_VERSION` is upgraded, since a future Besu release may fix
the layered-pool issue.

`blockchain/besu/qbftConfigFile.json` additionally pre-funds the well-known Hardhat/Anvil
local test account #0 (`0xf39Fd6e51aad88F6F4ce6aB8827279cffFb92266`, from the public devnet
mnemonic `test test test test test test test test test test test junk`) with a nominal
demo balance. This is a universally public, zero-value devnet key — never use it outside
local development, and never fund it on a real network. It is kept as a realistic stand-in
for the backend-controlled functional accounts (deployer, treasury, per-actor keystores)
that M5/M6 will create, since those will also hold a small balance rather than relying only
on `min-gas-price=0`.

## Commands

```sh
make besu-generate
make up
make network-status
make down
```

`make up` depends on generated Besu files, so it is safe to run from a clean checkout after Docker is available.

If `make besu-generate` reports that it cannot connect to the Docker daemon, open Docker Desktop and retry the same command.

## Validation checklist

After `make up`, run:

```sh
make network-status
```

Expected properties:

- `qbft_getValidatorsByBlockNumber` returns four validator addresses.
- `eth_blockNumber` increases over time.
- `eth_gasPrice` returns `0x0`.
- Only `rpcnode` publishes HTTP RPC to the host at `127.0.0.1:8545`.
